ภาพรวมระบบ (แยก 4 ส่วนชัด ๆ)
A) หน้าจอ TV (Display Player)

โหมดเต็มจอ (Chrome / Android TV / mini PC)

ดึง “คิว” จาก Server แบบเรียลไทม์ (WebSocket/SSE)

แสดง:

รูปผู้โดเนต + ชื่อ/ข้อความ

Tag (IG/FB/TikTok/เบอร์/อื่น ๆ)

QR กลาง “สแกนโดเนต”

เอฟเฟกต์ (intro animation / confetti / glow / sound เลือกได้)

B) หน้าโดเนต (Donor Page)

เลือกแพ็กเกจ 1/2/3/4 (ราคา + เวลาแสดง)

อัปโหลดรูป + ใส่แท็กติดต่อ + ข้อความสั้น

จ่ายเงิน → ระบบยืนยัน → เข้าคิวอัตโนมัติ

C) หลังบ้านร้าน (Admin/Staff)

ตั้งค่าร้าน: แพ็กเกจ, เวลาต่อแพ็ก, กฎข้อความ, keyword แบน, เปิด/ปิดฟีเจอร์พิกัด/รูป

คิวตอนนี้ / ถัดไป / ข้าม / แบน / อนุมัติ (ถ้าต้องการ moderation)

ดูรายได้รายวัน/รายเดือน/ตามช่องทาง

D) ระบบชำระเงิน + ตรวจสอบยอด

แนวที่ “งานโปรดักชัน” ควรใช้:

Payment Gateway ที่ให้สถานะชำระเงิน + webhook (ลดสลิปปลอม)

Opn Payments (Omise) รองรับ PromptPay

GB Prime Pay มีเอกสาร QR Cash (สร้าง PromptPay QR)

2C2P รองรับช่องทาง QR/PromptPay และหลายช่องทาง

ถ้าพี่อยาก “ตรวจสลิป” เพิ่ม: KBANK มี Slip Verification API docs (เหมาะเป็นตัวเลือกเสริม/เฉพาะเคส)


โครงคิว (หัวใจของระบบ)

ให้คิดคิวเป็น “Job” ที่มีเวลาแสดงชัดเจน

หลักการแนะนำ

ชำระเงินสำเร็จ → สร้าง queue_item สถานะ PENDING

(ถ้ามี moderation) แอดมินกดอนุมัติ → APPROVED

Display Player ดึงรายการถัดไป → PLAYING

ครบเวลา → DONE และเลื่อนไปตัวถัดไป

กันปัญหา “คนโดเนตถี่มาก”

ทำเป็น Priority by package ได้ (เช่น แพ็กใหญ่แทรกได้) หรือ

ทำเป็น Round-robin by amount (ยุติธรรม)
พี่เลือก policy ได้จาก setting

แพ็กเกจ 1–4 (ตัวอย่างที่ขายง่าย)

Package 1: 50฿ → แสดง 10 วิ

Package 2: 100฿ → แสดง 20 วิ

Package 3: 200฿ → แสดง 40 วิ

Package 4: 500฿ → แสดง 90 วิ + เอฟเฟกต์พิเศษ/เสียง

ทริค: ใส่ “สิทธิพิเศษ” ให้แพ็กบน เช่น glow frame / spotlight / ติด “TOP DONOR” 10 นาที
ทำให้คนอยากกดแพ็กสูงขึ้น

ความปลอดภัย/ความเสี่ยงที่ต้องใส่ตั้งแต่ MVP (ไม่งั้นร้านกลัว)
1) Moderation (แนะนำมาก)

เปิดโหมด “อนุมัติก่อนขึ้นจอ” ได้

blacklist คำหยาบ + เบอร์แปลก + ลิงก์

จำกัดตัวอักษร (เช่น 30–60 ตัว)

2) รูปภาพ

จำกัดประเภทไฟล์/ขนาด

ทำ watermark “Donated at <bar>”

เก็บไฟล์นอก webroot + random filename

3) Anti-spam

rate limit ต่อเบอร์/ต่อ IP

1 คนคิวพร้อมกันไม่เกิน N รายการ

4) PDPA แบบเบสิค

checkbox “ยินยอมแสดงรูป/ข้อมูลบนจอ”

ตั้งเวลาลบข้อมูลอัตโนมัติ (เช่น 30–90 วัน)

Database (แบบพอเริ่มทำได้เลย)

ตารางหลักที่พี่ต้องมี:

venues (ร้าน)

packages (1–4: ราคา/วินาที/สิทธิพิเศษ)

donations (order/payment status, amount, channel, ref)

media (image_url, thumbnail_url, approved_flag)

donor_profiles (display_name, message, ig, fb, tiktok, phone_masked)

queue_items (donation_id, venue_id, status, duration_sec, priority, scheduled_at, started_at, ended_at)

payment_events (webhook logs กันพลาด)

Tech stack ที่เข้ากับ Node + MySQL (ทำให้เสถียร)

API: Node (Express/Fastify)

Realtime ไปจอ: WebSocket หรือ SSE

Queue worker: cron/worker (เช่น BullMQ + Redis ถ้ามี) หรือทำใน DB ก็ได้ตอน MVP

Display: Web app fullscreen (Next.js/Vite ก็ได้) เปิดบน TV

Upload: S3-compatible (หรือ local storage ถ้าร้านเดียวก่อน)
